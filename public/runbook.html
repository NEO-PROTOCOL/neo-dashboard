Quando usar o Analyzer

Antes de mudar prioridade, antes de discutir incidentes recorrentes, antes de chamar algo de “urgente” sem lastro estrutural.

Quando usar o Control

Durante investigação operacional, leitura de logs, runtime health, filas, eventos e ações autenticadas.

Quando usar a Topology

Para entender relações, nós desconectados, leitura visual da malha e contexto de dependência. Não use como painel primário de decisão.

Operador

Objetivo: detectar prioridade operacional sem se perder em instrumentação decorativa.

Fluxo diário

Abra /stack-analyzer.html, valide readiness, precision e findings. Só depois entre em / para ações live.

Sinais que importam

High findings, DNS pendente, ausência de healthcheck, conflitos de porta e serviços sem produção.

Erro comum

Confundir animação com observabilidade. Se a tela brilha mas não explica risco, ela está te cobrando atenção sem pagar inteligência.

DevOps

Objetivo: reduzir tempo entre sintoma, hipótese e ação corretiva.

Fluxo de incidente

Comece em /api/monitor/health autenticado, passe para / para logs e eventos, use /ecosystem-3d.html somente para contexto topológico.

Leitura do grafo

Se o modo estiver Snapshot, não trate alertas visuais como verdade operacional. Live é estado, snapshot é memória.

Regra de ouro

API protegida permanece protegida. Se um dado precisa ser público, crie uma superfície pública específica. Nunca desproteja nervo porque a interface ficou preguiçosa.

Arquitetura

Objetivo: governar a malha sem misturar runway visual com governança técnica.

Uso principal

Leia o Analyzer para decidir backlog estrutural, debt prioritário e pontos de fragilidade sistêmica entre organizações e nós.

Perguntas corretas

Qual nó concentra risco? Onde falta healthcheck? Onde a topologia mascara dependência? Onde o deploy existe sem reproducibilidade?

Critério

Se uma superfície não deixa claro se mostra estado live, snapshot ou leitura derivada, ela precisa ser reformada antes de ser confiada.
